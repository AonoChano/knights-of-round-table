## 📘 Anthropic Skill 完全指南：大模型时代的“能力模块化”革命

### 一、什么是 Skill？

Anthropic Skill（又称 Agent Skill）是 Anthropic 于 2025 年 10 月推出的一种**模块化、可复用的大模型能力扩展机制**。它将特定领域的专业知识、标准流程和操作规范封装成可被模型自动理解和执行的“能力单元”。

通俗来说，Skill 就像给 AI 准备的 **“专家级操作手册”或“工作交接 SOP 大礼包”** —— 当你要把一项工作交给一个新同事时，你不会只靠口口相传，而是会准备一套完整的文档，告诉他：“遇到这类情况，按以下步骤处理”。

**核心特性一览：**

- **模块化封装**：将复杂任务的知识、步骤和输出规范固化为独立单元，便于复用和维护
- **自动触发**：模型根据上下文语义自动发现并调用合适的 Skill，无需用户显式指令
- **结构化约束**：通过明确的执行步骤和输出格式，显著降低模型“自由发挥”的不确定性
- **轻量可扩展**：Skill 可仅包含指令，也可附带模板、示例或脚本资源

**关键时间线**：2025 年 10 月，Anthropic 首次发布 Claude Skills；同年 12 月 18 日，将其作为**开放标准**进一步发布，推动 AI Agent 开发生态走向标准化。


### 二、核心运作机制：三层渐进式披露

Skill 最具巧思的设计是 **“渐进式披露”** —— 技能信息分为三个层次，模型按需逐步加载，既确保细节完整，又避免一次性塞入过多内容。

| 层级 | 内容 | 加载时机 | Token 占用 |
|---|---|---|---|
| 第一层：元数据 | 技能名称 + 简短描述（SKILL.md 的 YAML 头部） | 智能体启动时自动预加载 | 极低（~20-30 token/个） |
| 第二层：技能主体 | 完整的 SKILL.md 详细指令、工作流说明 | 模型判断技能与当前任务相关时加载 | 适中（建议 <5K token） |
| 第三层：附加资源 | 脚本（Python/Shell/JS）、模板、参考文档 | 仅当需要执行具体操作时才加载 | 无限制 |

**实际运行流程：**

1. 启动时，系统扫描技能目录，**只读取每个 SKILL.md 的元数据**（name + description），并将其注入系统提示词
2. 用户输入任务后，模型根据元数据判断匹配的技能
3. 一旦决定调用某技能，系统才**动态加载完整的 SKILL.md 内容**到对话上下文
4. 根据 SKILL.md 的指引，模型逐步执行任务，必要时调用附加脚本

这种设计的精妙之处在于：即使安装了 100 个不同的 Skills，系统启动时也只需处理几千个 token 的元数据，而不是把所有技能的完整内容都加载进来。


### 三、Skill 的技术架构

#### 3.1 标准文件夹结构

```
my-skill/                    # 技能名称（与文件夹名一致）
├── SKILL.md                # 必选：核心技能定义文件
├── scripts/                # 可选：可执行脚本（Python/Shell/JS 等）
├── templates/              # 可选：模板文件
├── resources/              # 可选：参考材料、数据文件
└── reference.md            # 可选：详细参考文档
```

#### 3.2 SKILL.md 文件格式

```markdown
---
name: meeting-summarizer
description: 将会议内容按「参会人员、议题、决定、要点」四要素进行结构化总结
---

# 会议总结专家

## 触发条件
当用户输入包含“会议总结”、“帮我总结这段会议”等关键词时激活。

## 执行步骤
1. 识别参会人员（从对话中的姓名提及）
2. 提取讨论议题
3. 归纳最终决定
4. 总结核心要点

## 输出格式
- 参会人员：XXX
- 议题：XXX
- 决定：XXX
- 要点：XXX

## 注意事项
- 不得添加原文没有的信息
- 保持客观，不做主观评价
```
- **YAML 前置元数据**：name（必填）和 description（必填），用于模型在启动时识别技能用途
- **Markdown 正文**：提供详细的指令、执行流程、示例和约束条件
- **YAML 头与 Markdown 正文之间必须空一行**


### 四、Skill 的工程边界：Tool / Skill / Agent 三层架构

| 层级 | 角色 | 职责 | 示例 |
|---|---|---|---|
| **Tool（操作层）** | 原子操作 | 执行具体动作，不包含业务逻辑 | 读取文件、执行 SQL、发送 HTTP 请求 |
| **Skill（能力层）** | 能力封装 | 封装完整任务的方法论、流程和约束 | “按品牌规范生成 PPT”、“分析财报并生成摘要” |
| **Agent（调度层）** | 智能调度 | 理解用户意图，决策调用哪个 Skill | Claude 主模型，判断用户需求并激活相应技能 |

> **关键理解**：Agent 不直接操作底层工具，Skill 负责封装“怎么做”的能力语义，Tool 执行具体动作。当 Skill 清晰时，Agent 更简单；当 Skill 混乱时，调度逻辑会指数级膨胀。


### 五、与传统方式的对比

| 维度 | **Prompt（提示词）** | **MCP（模型上下文协议）** | **Tool/Function Calling** | **Skill** |
|---|---|---|---|---|
| **本质** | 单次对话的文本指令 | 连接外部数据/工具的协议 | 模型输出结构化数据触发外部代码 | 可持久化、可复用能力单元 |
| **复用性** | 每个对话需重新编写 | 工具可复用，但无流程封装 | 函数本身可复用 | 标准化封装，跨项目多场景复用 |
| **加载机制** | 全量载入上下文 | 启动时加载所有工具定义 | 函数定义常驻上下文 | 延迟加载，按需激活完整内容 |
| **适用场景** | 临时性、一次性任务 | 需要访问实时数据/外部系统 | 单一原子操作 | 固化专业流程、团队最佳实践 |

- **Skill vs Prompt**：Prompt 适合单次任务，Skill 才是构建可复用 AI 能力的正确做法。Prompt 随对话消失，Skill 可被不同对话中的模型发现和复用
- **Skill vs MCP**：MCP 解决“模型能用到什么”（工具和数据接入），Skill 解决“模型该怎么用”（流程和方法论）。两者是**协作而非竞争**关系——Skill 教会模型如何正确使用 MCP 提供的工具
- **Skill vs Function Calling**：Function Calling 只是模型执行某些动作时可能用到的底层手段，不是 Skill 的定义层。Skill 的核心机制是**上下文注入**——模型读取 Markdown 文档，把其中的规则和流程纳入推理上下文


### 六、Skill 的核心优势

#### 6.1 Token 效率革命
传统方案中，大量工具定义和指令必须常驻上下文窗口，极易超出限制。Skill 通过三层渐进式加载，实现按需激活：元数据保持轻量（~30 token/个），只有真正需要的技能才加载完整内容。这使得模型可以拥有海量专业知识，却几乎不额外占用上下文窗口。

#### 6.2 能力沉淀与团队复用
重复逻辑不再写成 Prompt，而是封装成 Skill。团队的最佳实践、合规流程、品牌规范可打包为技能，一键分发给所有同事的 Agent。Skill 因此成为组织知识沉淀的理想载体。

#### 6.3 质量可控
通过明确的输入假设、执行步骤和输出格式，Skill 显著降低了模型“自由发挥”的不确定性。在架构层面，Skill 使 AI 系统具备**可观测、可追踪的能力链路**：决策路径可记录、工具调用可日志、异常路径可覆盖、重试回滚有机制。

#### 6.4 开放生态
2025 年 12 月 18 日，Anthropic 将 Agent Skills 发布为**开放标准**，Microsoft（VS Code、GitHub）、OpenAI（ChatGPT、Codex CLI）、Cursor 等均已跟进。一个 Skill 写好之后，可以在任何支持该标准的 Agent 平台上运行。

### 八、如何使用与开发 Skill

#### 8.1 使用方式

Skill 是**自动激活**的，用户只需正常描述需求，模型会自动判断并调用相应技能：

- “帮我提取这份合同里所有涉及付款的条款”（自动调用 PDF Skill）
- “分析这份电商销售数据，找出 GMV 最高的品类趋势”（自动调用 data-analysis Skill）
- “创建一个自动生成周报的插件”（自动调用 skill-creator Skill）

#### 8.2 快速安装体验

```bash
# 1. 注册官方 Skill 仓库为插件源
/plugin marketplace add anthropics/skills

# 2. 安装文档处理包（含 pdf、xlsx、docx、pptx）
/plugin install document-skills@anthropic-agent-skills

# 3. 安装创意和开发包（含 frontend-design、skill-creator 等 12 个技能）
/plugin install example-skills@anthropic-agent-skills
```

#### 8.3 开发一个自定义 Skill（3 步）

```bash
# 步骤 1：创建目录结构
mkdir -p ~/.claude/skills/my-skill
cd ~/.claude/skills/my-skill

# 步骤 2：编写 SKILL.md（如上文示例格式）

# 步骤 3：重启即生效，无需额外配置
```

**开发最佳实践**：
- 元数据中的 description 要足够清晰，让模型知道何时使用
- SKILL.md 正文建议控制在 5K token 以内
- 使用 `allowed-tools` 字段明确声明所需权限，增强安全性
- 可配合 skill-creator 辅助生成初始模板


### 九、当前挑战与发展趋势

- **自动生成的 Skill 质量不足**：Anthropic 自带的 skill-creator 上线第一周，开发者观察 100 多个用户的使用情况后指出，大多数自动生成的实现“看起来更像玩具而不是工具”——触发不准确、指令过多导致模型混乱、安全漏洞等问题反复出现
- **Skill 正在向 Agent 自主编写演进**：行业正在探索让 Agent 能够自己发现解决问题的方法，并自动生成高质量 Skill
- **从“个人技巧”到“工程化”**：Skill 的出现，标志着大模型能力正在从规模竞争、推理能力竞争，进入**第三阶段：能力工程化**。真正的差距将不再体现在模型参数上，而在于能力结构的设计


### 十、总结

**一句话概括：Skill = SOP × AI。**

Anthropic Skill 不只是另一个“插件”或“工具集”，它是大模型进入结构化时代的关键标志——将“如何把事做对”从 Prompt 技巧提升为可工程化、可复用、可共享的能力单元。无论你是希望固化团队工作流的企业用户，还是希望提升个人 AI 使用效率的个体用户，Skill 都是当前最值得投入理解与实践的 AI 能力范式之一。
